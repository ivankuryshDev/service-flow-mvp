Build ServiceFlow as a portfolio app.

A single source-of-truth spec for implementing the ServiceFlow booking platform as a frontend-only portfolio project in React / Next.js (App Router) + TypeScript + Tailwind. It reconciles the design system, the Booking App UI kit, and the project screens into one final inventory, then defines routes, tokens, components, interactions, and exactly what ships in the first demo.

Next.js App Router TypeScript Tailwind CSS Frontend-only · mock data No backend · no real payments

Read-only  How to use this document

This is documentation only. It references the existing design system and screens but changes none of them — treat the design system, the Booking App kit, and the project screens as frozen inputs. When a control looks clickable, this doc says what it should do; anything not in MVP scope is marked Coming soon or disabled rather than left ambiguous.

1 · Final screen inventory

ServiceFlow splits into two surfaces: a customer booking flow (search → choose → select time → confirm) and a business owner console (dashboard, bookings, availability). The customer flow lives in the design system's Booking App kit; the deeper, production-ready versions of two of those steps — plus the entire business console — live in the project screens.

Duplicates & conflicts — how to resolve

  • Service details & Confirmation appear twice. The Booking App kit is a cosmetic click-through; the project screens are the production build with real states. Treat the project versions as final for these two and retire the kit's inline copies.
  • Locale / currency conflict (important). The customer flow is set in Brooklyn, NY with USD (e.g. Deep Home Clean, $130); the business console is set in Dublin, IE with EUR (Dublin Laptop Care, €140). For a coherent demo, pick one locale and currency and apply it across both surfaces, or frame them as two seeded accounts. Recommendation: standardize on one (e.g. Brooklyn / USD) and drive currency from a single config constant.
  • Provider identity differs by surface. Customer side = "Tidy & Co." cleaning; business side = "Dublin Laptop Care" repairs. Fine as separate seed personas, but the demo reads cleaner if the confirmed customer booking and a business booking reference the same service vocabulary.

Not in scope to build now

Auth/login, provider onboarding ("List your service"), Favorites, Messaging, Calendar (month) view, Services catalog editor, and real payments are referenced by navigation but have no designed screen. They stay as Coming soon destinations — see §7.

3 · Design tokens for implementation

Every value below is already a CSS custom property in the design system's tokens/ files. Wire them into tailwind.config.ts as var(--*) references (the design system README ships the exact mapping). Build against tokens and DS classes — never hardcode hexes in components.

Colors

A warm triad — sunny yellow (the sun / primary action), warm charcoal ink (type + secondary), airy cream surfaces. No cold greys. Blue is an info accent only (reminders) — never a primary button. Primary buttons are always yellow fill with charcoal text.

  • Mono — DM Mono for times, prices, and confirmation codes (ticket-stub feel).

Casing: sentence case everywhere. Uppercase only for wide-tracked eyebrow/overline labels (--tracking-wide 0.04em). No emoji in product copy.

Shadows

Soft, diffuse, warm-tinted (brown-black rgba(43,36,20,…)) — never hard black. A special yellow glow sits under primary CTAs and selected slots.

  • --shadow-sm — cards at rest · --shadow-md — raised panels
  • --shadow-lg — card hover (pair with a −4px lift) · --shadow-xl — modals
  • --shadow-accent = 0 10px 28px rgba(245,176,0,0.32) — the yellow CTA glow

Button styles

Button ships four variants × three sizes; pill-shaped, touch-friendly.

  • primary — yellow-400 fill, charcoal (ink-900) text, accent glow. Hover darkens 500→600; press nudges down + scales ~0.985.
  • secondary — solid charcoal fill, cream text.
  • outline — transparent, warm border; border darkens to ink on hover.
  • ghost — no border; fill warms to cream on hover.
  • Sizes: sm 40px · md 52px · lg 62px. Props: full, leadingIcon, trailingIcon, disabled.
  • Never white-on-yellow or a blue primary.

Card styles

  • Base card — white fill, soft warm hairline (--border-soft), shadow-sm at rest, shadow-lg + −4px lift on hover, radius-lg.
  • ServiceCard — photo-led; big rounded image (warm yellow gradient fallback) with category pill + duration chip overlaid; rating star is the one filled glyph (yellow-500).
  • BookingCard — boarding-pass: yellow date block + dashed tear line.
  • Avoid the SaaS "rounded box with a colored left-border accent" trope. The only left rails used are the Toast's semantic rail and the BookingCard's yellow date panel.

Form styles

  • Inputs ≥ 56px tall (lg up to 66px); radius-md (14px); warm --border-input hairline.
  • Focus — border swaps to yellow-500 + 4px soft-yellow glow ring.
  • Error — error-500 border + helper text below; Input takes an error prop.
  • Disabled — muted, no shadow, not focusable.
  • Components: Input (with optional leading icon), Select (custom chevron), SearchField (signature booking bar), DatePicker (day strip), TimeSlot (available / selected / unavailable). Touch targets never below 44px.

5 · Interaction specification

Per screen: clickable elements and expected behaviour, plus the state matrix (loading / empty / error / disabled) the build must handle. States already present in the project screens are noted as designed; the rest follow the design system's empty/error patterns. Anything outside MVP is marked Coming soon — disable it, don't hide it.

Home / Search /

Clickable elements & behaviour

  • Service select + location input + date strip → build the search query.
  • "Find services" (primary) → navigate to /search with the chosen filters in the query string.
  • Category tiles → /search?cat=….
  • TopNav links Explore (active) · Bookings/Favorites Coming soon.
  • "List your service" CTA → Coming soon.
  • State changeControlled inputs update local search state; DatePicker highlights the selected day.
  • LoadingStatic page — none needed on load.
  • EmptyN/A (categories are fixed seed data).
  • ErrorN/A in MVP.
  • Disabled"Find services" stays enabled (location pre-filled); disable only if location is cleared.

Search Results /search

Clickable elements & behaviour

  • Category checkboxes → toggle filter; results recompute live.
  • Price slider → cap max price live. Rating & Availability pills → single-select filter.
  • Clear (n) → reset all filters (shown only when filters are active).
  • Sort select → Recommended / Price low→high / Top rated.
  • Result card "View availability" → /services/[serviceId].
  • "Edit search" → back to Home (or reopens the search bar).
  • State changeFilters/sort drive the visible list and the "N services available" count; sync to URL query.
  • LoadingSkeleton result cards while results resolve (MVP: instant from mock; keep a brief skeleton for realism).
  • EmptyDesigned — EmptyState "No services match those filters" + "Clear all filters" action.
  • ErrorReuse the console error pattern (icon + retry) if a future fetch fails; N/A for mock.
  • Disabled"Clear" hidden when no filter is active.

Service Details & Booking /services/[serviceId]

Clickable elements & behaviour

  • Breadcrumb / "Back to results" → /search.
  • DatePicker day → selects day, clears chosen slot, triggers a short slot reload.
  • TimeSlot → selects time (unavailable slots not clickable); updates the summary panel + total label.
  • "See next available" (fully-booked day) → jumps to the next day with openings.
  • Name / contact / note inputs → controlled; clearing a field's value clears its error.
  • "Confirm booking" (primary) → validates; on success opens the confirm Modal; on failure fires an error Toast + inline messages.
  • Modal "Confirm & pay" → finalizes: closes modal, sets confirmed, success Toast, scrolls to top. In the real app this routes to /booking/[ref]/confirmation.
  • Modal "Back" / close → dismiss, keep form state.
  • State changeday · slot · form fields · errors · modal open · confirmed · toast — all designed in the screen.
  • LoadingDesigned — slot grid shows skeleton cells (~650ms) after changing day.
  • EmptyDesigned — fully-booked day renders EmptyState "No openings that day".
  • ErrorDesigned (validation) — missing name / contact / slot → inline field errors + summary error + error Toast. Add a slot-load error fallback if wired to a fetch.
  • DisabledDesigned — after confirm, CTA becomes a disabled "Booking confirmed". Unavailable TimeSlots are non-interactive.

6 · MVP scope — the first portfolio demo

The demo should let a viewer walk both stories end-to-end on mock data, with every designed state reachable. No backend, no auth, no payments — persistence is in-memory (React state) and resets on refresh.

Customer story — must work

  • Home → enter a search → Search Results with working filters, sort, and the designed empty state.
  • Open a result → Service Details: pick a day (with slot loading), pick a time, fill details, hit validation, confirm via modal.
  • Land on Confirmation showing the booked details; toggle/observe confirmed vs pending.

Business story — must work

  • Dashboard: stats, today's bookings with tab filtering + per-row confirm, and the new-business empty account.
  • Bookings Management: search + filter + tabs, the detail drawer with status-aware actions, and all four table states (data / loading / empty / error).
  • Availability: weekly board, add/edit slot modal, block/clear/copy quick actions, overlap warning, loading + error states.

Cross-cutting — must work

  • Both TopNavs with correct active states; responsive down to tablet; full design-token fidelity (yellow primary, warm neutrals, pill shapes, soft shadows).
  • Toasts (auto-dismiss), Modals (scrim + spring), and all empty/loading/error patterns from the design system.
  • Keyboard focus rings (yellow glow), ≥44px touch targets, sentence-case copy, no emoji.

Definition of done

A reviewer can complete a booking as a customer and manage it as a business owner, every interactive control either does something or is clearly marked Coming soon, and no state in this spec is unreachable.
